## 三、未映射或映射不全的字段

### 1. status（当前房源状态）【高优先级】

- **含义**：`available`=可租，`rented`=已租出
- **用例**：无显式「只要在租的」表述，但默认语义是只展示可租房源
- **现状**：未在搜索/筛选层过滤 status，若 API 返回已租房源会一并展示
- **建议**：在 `get_houses_by_platform` / `get_houses_nearby` 参数或 filter 层增加 `status=available`，或在 `_house_matches_spec` 中默认排除 `status=rented`

---

### 2. hidden_noise_level 扩展【中优先级】

- **取值**：安静、中等、吵闹、临街
- **用例**：
  - EV-008: 睡眠不好，需要环境安静
  - EV-012: 睡眠不好，需要环境安静的房子
  - EV-015: 我睡眠浅，需要安静一点的
  - EV-035: 一定要安静，不能吵
  - EV-061: 希望环境能安静一些
  - EV-077: 我睡眠比较浅，所以对噪音比较敏感，希望能找个安静的房子
  - EV-088: 隔音一定要好，怕吵
  - EV-098: 在家工作要环境安静，不能吵
  - EV-100: 隔音一定要好，楼上楼下不能吵
  - EV-122: 不要吵闹不隔音的房子
- **现状**：
  - 已有：`"安静"` → `["hidden_noise_level", "安静"]` 正向筛选
  - 已有：安静/隔音/睡眠浅/怕吵/不能吵/不隔音 → 排除 `hidden_noise_level=吵闹`
- **缺失**：
  - 「环境安静」「不能吵」「隔音好」「噪音敏感」等未统一映射
  - 「临街」未处理：用户要安静时应排除 `hidden_noise_level=临街`

**建议补充**：
```
"环境安静" → ["hidden_noise_level", "安静"]
"不能吵" → ["hidden_noise_level", "安静"] 或 排除 吵闹
"隔音好" → 排除 hidden_noise_level 吵闹、临街
"噪音敏感" → 排除 吵闹、临街
```

并在 _EXCLUDE_RULES 中增加：要安静 → 排除 `hidden_noise_level=临街`

---

### 3. floor（楼层）【中优先级】

- **取值**：低层、中层、高层、共1层、共5层 等
- **用例**：
  - EV-019: 希望是一楼或者有电梯的房子
  - EV-105: 最好还是高层，视野会好一些
- **现状**：未映射 floor
- **建议**：
  - 「一楼」：`floor` 含「低层」或 `total_floors` 较小，或新字段 `floor_level=1`
  - 「高层」：`floor` 含「高层」，或与 `total_floors` 组合判断

---

### 4. available_from（可入住日期）【中优先级】

- **含义**：可入住日期
- **用例**：
  - EV-015: 我想3月10号前入住
  - EV-066: 希望3月10日前能入住
- **现状**：未映射
- **建议**：解析日期（如「3月10号前」）→ `available_from <= 2026-03-10`，需确认 API 是否支持该参数

---

### 5. property_type（物业类型）【低优先级】

- **取值**：住宅、公寓
- **用例**：暂无明确「只要住宅」或「只要公寓」的表述
- **现状**：未映射
- **建议**：若有需求再补充，如「公寓」→ property_type=公寓

---

### 6. area_sqm（面积）【低优先级】

- **用例**：
  - EV-122: 房间要大一点，能放下书桌和电脑
  - EV-059: 想租个客厅大一点的两居室
- **现状**：首轮有 `min_area` 解析，但多轮 filter 无 area_sqm 条件
- **建议**：在 _FILTER_KEYWORDS 增加「房间大」「面积大」→ `min_area` 或 `area_sqm >= 某值`

## 五、建议补充的 _FILTER_KEYWORDS 与 _EXCLUDE_RULES

### 1. hidden_noise_level 增强

```python
# _FILTER_KEYWORDS 补充
"环境安静": ["hidden_noise_level", "安静"],
"不能吵": ["hidden_noise_level", "安静"],
"隔音好": ["hidden_noise_level", "安静"],
"噪音敏感": ["hidden_noise_level", "安静"],
"睡眠质量差": ["hidden_noise_level", "安静"],

# _EXCLUDE_RULES 补充
(["安静", "环境安静", "睡眠浅", "怕吵", "隔音"], "hidden_noise_level", "临街"),
```

### 2. status 过滤（需确认 API 支持情况）

- 方案 A：API 调用时传 `status=available`
- 方案 B：`_house_matches_spec` 或结果后处理中排除 `status=rented`
- 方案 C：在 rental_tools / API 层统一只返回可租房源

### 3. available_from（可入住日期）

- 解析「3月10号前」「3月10日前」→ 提取日期
- 若 API 支持 `available_from` 或 `available_before`，则传入

### 4. floor（楼层）

- 「一楼」：可映射为 floor 含「低层」或楼层=1（若数据结构支持）
- 「高层」：floor 含「高层」

## 六、总结表

| 字段 | 是否映射 | 说明 |
|------|----------|------|
| status | ❌ | 应过滤 rented，仅展示 available |
| hidden_noise_level | ⚠️ 部分 | 安静/吵闹已覆盖，缺环境安静/隔音/临街排除 |
| floor | ❌ | 一楼、高层等未映射 |
| available_from | ❌ | 可入住日期未解析 |
| property_type | ❌ | 住宅/公寓未区分 |
| area_sqm | ⚠️ 部分 | 首轮有 min_area，多轮 filter 无 |

**优先建议**：  
1. 实现 status 过滤，避免展示已租房源。  
2. 完善 hidden_noise_level 映射并补充临街排除。  
3. 视 API 能力补充 available_from、floor 的解析与传递。
