## 二、关键差异分析

### 2.1 响应格式差异（高优先级）

**正确输出** 格式示例：
```
为您找到5套符合条件的房源：
1. 万科居9区 | 2800元/月 | 精装 | 地铁323m | 中石化 | 合租
2. 金地苑9区 | 2850元/月 | 精装 | 地铁493m | 国家体育 | 合租
...
```

**当前模型** 格式：
```json
{"message": "为您找到5套符合条件的房源", "houses": ["HF_5611", "HF_1291", ...]}
```

**建议**：在保留 `message` + `houses` JSON 结构的前提下，将 `message` 扩展为带房源详情的可读文本，以满足评测或展示需求。

### 2.3 根因归纳

1. **响应格式**：缺少房源详情展示，仅返回 ID。
2. **链家/58 平台**：`listing_platform` 指定链家时 API 无数据，需更早触发安居客回退。
3. **复杂条件查询**：24h餐饮、健身房、附近地标等依赖 `search_landmarks` + `get_houses_nearby`，可能未正确串联。
4. **Tag 映射不全**：仓鼠、采光/有阳光、地下车库等需补充到 `_FILTER_KEYWORDS`。
5. **orientation 参数**：`有阳光` 非有效 API 值，应映射为 `采光好` tag 或 `朝南`。
6. **会话继承**：部分多轮场景正确输出未继承上一轮 houses，反而重新搜索或闲聊。

## 三、重构建议（按优先级）

### P0：响应格式增强

- 从 tool 返回中解析房源字段（community, price, decoration, subway_distance, subway_station, rental_type）。
- 在 `message` 中生成：`N. 小区名 | 价格元/月 | 装修 | 地铁Xm | 地标 | 租住类型`。
- 保持 `houses` 数组与 `message` 中顺序一致。

### P1：平台与地标回退

- 链家/58 无结果时，立即用安居客重试（已有逻辑，需确保覆盖所有入口）。
- 对「24小时餐饮」「健身房」等，优先用 `search_landmarks`（餐饮/健身房）配合 `get_houses_nearby`，再与基础条件做交集。

### P2：Tag 与过滤扩展

- 仓鼠 → `可养宠物`。
- 有阳光/采光好 → `采光好` tag 或 `orientation=朝南`。
- 地下车库/地库 → `车库车位` 或 `近车库` 等 tag。
- 补充：24小时餐饮、健身房、菜市场、医院、公园等在 `_FILTER_KEYWORDS` 中的映射。

### P3：Prompt 与工具说明

- 明确要求：有候选房源时，追加条件必须用 `get_house_by_id` 过滤，禁止重新 `get_houses_by_platform`。
- 在 tools schema 中说明 `orientation` 合法值（朝南、南北等），避免传 `有阳光`。

### P4：兜底策略

- 当多条件过滤结果为空时，可考虑「放宽最不重要条件」或给出明确说明，避免频繁「查询失败，请重试」。
